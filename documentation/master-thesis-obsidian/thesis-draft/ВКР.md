# ВКР. Автоматизация отбора кандидатов на основе обработки естественного языка

Данный документ является консолидированной версией доступных разделов выпускной квалификационной работы, размещенных в репозитории. Разделы, связанные со сравнением метрик и результатов модели, синхронизированы с обновленными формулировками пояснительной записки.

## 2.1 Разработка математической модели отбора кандидатов на основе обработки естественного языка из резюме и вакансий

Разработка математической модели для автоматизированного отбора кандидатов на основе обработки естественного языка из резюме и вакансий представляет собой комплексную задачу, включающую несколько ключевых этапов: извлечение информации, ее семантическое сопоставление и принятие решений. Основной целью является создание модели, способной анализировать текстовые данные, выделять ключевые параметры и сравнивать их с требованиями вакансий.

Один из возможных подходов включает использование трансформерных моделей, таких как BERT, которые позволяют учитывать контекст слов в предложении. Модель BERT представляет текст в виде токенов и создает их контекстуальные представления, учитывая как предыдущие, так и последующие слова в предложении. Это создает предпосылки для более точного сопоставления резюме и вакансий по сравнению с методами, основанными только на совпадении ключевых слов, так как BERT позволяет различать значения одного и того же слова в зависимости от контекста. Фактическая величина улучшения должна определяться экспериментально относительно обоснованных контрольных подходов. Например, фраза «работал в команде» будет иметь различные векторные представления в зависимости от общего контекста резюме [6].

Таким образом, разработка математической модели для автоматизированного отбора кандидатов требует использования методов обработки естественного языка, таких как трансформеры, для извлечения смысловой информации из текстов резюме и вакансий. Модель должна учитывать контекстуальные зависимости между словами.

У нас есть два множества объектов: резюме и вакансии. При этом у каждого из них есть определенный набор свойств, обозначим их следующим образом:

Vv1,...,vn (1)

Формула 1. Вакансии

Rr1,...,rk (2)

Формула 2. Резюме

Из предметной области поиска работы и сотрудников известно, что как у вакансий, так и у резюме может быть неограниченное число параметров n и k, но не менее одного параметра для каждого объекта. Например, параметр «название» может принимать значение «Продуктовый аналитик» и для резюме, и для вакансии. У вакансии и у резюме могут быть параметры различных типов: число, строка, булевое значение и другие. При выборе конкретного однотипного набора вакансий типы для всех параметров должны быть одинаковыми. Поскольку резюме и вакансия отображают некоторое множество объектов профессий, могут быть сделаны преобразования, позволяющие определить функции вида F: R→V и F-1: V→R.

Pp1,...,pn (3)

Формула 3. Профессии

Специалист службы кадров при составлении вакансии реализует функцию вида G: P→V, которая является проекцией множества профессий на множество вакансий. В свою очередь соискатель при составлении резюме может осуществлять отображение A: P→R на основе предпочитаемой профессии. Путем обратных преобразований вакансий и резюме можно получить элементы из множества «Профессии». При этом работодатель может искать соискателя не только на основании профессиональных качеств, но и на основании других признаков. Поэтому множество P может быть дополнено набором признаков, характеризующих профессионала.

Ниже представлено преобразование, которое позволяет найти отображение множества резюме на множество вакансий:

V*=GA-1R, V*∈V (4)

Формула 4. Переход множества резюме в множество вакансий

Следует учитывать, что множество V* не обязано быть эквивалентным множеству V. Также не все соискатели составляют резюме на основе желаемой профессии, а значит существуют случаи, когда резюме не может быть корректно отображено в множество «Профессионалы».

Специалист службы кадров в процессе своей деятельности сортирует пришедшие резюме на вакансию. Этот процесс представляет собой последовательность действий, где для каждого резюме проставляется статус «принято» либо «отклонено». Тогда требуемую модель можно представить так:

M2(V,R)[0,1] (5)

Формула 5. Модель номер два осуществляет переход пары вакансия-резюме в отрезок от нуля до единицы.

Данный отрезок связан со степенью уверенности, куда отнести конкретное резюме для конкретной вакансии. Значение 0 означает, что резюме на данную вакансию моделью отклонено со 100% уверенностью, а значение 1 означает, что резюме принято со 100% уверенностью. Значение 0.5 означает, что резюме принято с вероятностью 50%.

В терминах предварительных рассуждений такая модель реализует функцию кадровика: отображать множество «резюме» на множество «профессионалы» и оценивать, насколько данный конкретный профессионал, представленный резюме, схож с профессионалом, который был отображен на множество вакансий.

F(A-1(R), G-1(V))[0,1] (6)

Формула 6. Функция F реализует отображение двух элементов из множества P в отрезок между нулем и единицей.

Вторая модель аппроксимирует некоторую функцию F*=(6). Для аппроксимации F* применяется BERT. Резюме и вакансии представлены преимущественно в виде текста; их текстовую часть обозначим как Rt и Vt.

Трансформеры используют механизм внимания для вычисления весов между токенами текста, позволяя модели учитывать контекст как слева, так и справа. Для задачи анализа вакансий и резюме модель BERT используется в качестве эмбеддинга текста E(R) и E(V) для резюме R и вакансий V соответственно. Полученные эмбеддинги можно использовать для вычисления сходства, например через косинусное расстояние. В качестве целевой функции для обучения модели классификации на основе вероятности принятия резюме используется кросс-энтропия.

## 2.2 Эксперименты с моделью на тестовых данных

Алгоритм сбора датасета для автоматизации отбора кандидатов предполагает последовательное извлечение, обработку и структурирование данных из резюме и вакансий. Первым этапом осуществляется сбор данных через платформы, где соискатели подают резюме на вакансии, а HR-специалисты принимают решения о приглашении или отказе. На втором этапе анализируются события, такие как «приглашен» или «отклонен», чтобы выделить уникальные пары «резюме-вакансия» и устранить дубли, где одно резюме попадало в обе группы. Далее данные очищаются от пропусков и получают актуальные на момент отклика заголовки вакансий и резюме. Завершается процесс преобразованием данных в формат, подходящий для машинного обучения, с учетом баланса классов и конфиденциальности информации.

Соискатели присылают свои резюме на вакансии, после чего HR-специалист может принять, отклонить или оставить отклик без действия. Рассмотрим только случаи, где HR-специалист принял решение, так как при отсутствии действия нельзя однозначно определить, подходит ли данное резюме для вакансии.

Далее был проведен этап токенизации с использованием предобученного токенизатора модели BERT. Текстовые данные были преобразованы в числовой формат, включающий индексы токенов, маски внимания и сегментные индикаторы. После этого данные были разделены на батчи для более эффективной обработки на GPU.

На этапе обучения была инициализирована модель BERT с головой классификации. Обучение проводилось с использованием оптимизатора AdamW и планировщика обучения с линейным уменьшением скорости обучения. Для контроля переобучения использовались метрики точности на валидационном наборе данных. После завершения обучения качество модели оценивалось на тестовых данных по метрикам accuracy и MCC; сравнение выполнялось не с искусственно заданным порогом, а с контрольными точками, связанными со структурой датасета и свойствами выбранных метрик.

По графику обучения видно, что признаки переобучения появляются после второй эпохи. Поэтому для рассматриваемого эксперимента две эпохи можно считать рациональным выбором, однако этот вывод ограничен текущим разбиением данных и должен уточняться при повторных экспериментах.

В итоге получена модель, для которой accuracy на задаче бинарной классификации составляет 0.75. Относительно majority-class baseline, равного 0.56 для итогового распределения классов, результат выше на 0.19 абсолютного значения метрики. Такое сравнение является более корректным, чем сопоставление с произвольным baseline, так как контрольная точка 0.56 отражает качество простого подхода, использующего только распределение классов и не использующего содержательные признаки текста.

Также в качестве метрики был взят коэффициент корреляции Мэтьюса (Matthews correlation coefficient, сокращенно MCC [29]). Данная метрика используется, так как в данных присутствует дисбаланс классов. Получаем MCC на всей выборке, равный 0.550. Поскольку MCC принимает значения от -1 до 1, где 0 соответствует отсутствию информативной связи между предсказаниями и истинными метками, значение 0.550 указывает на наличие полезного сигнала в модели. В то же время этот результат не следует интерпретировать как исчерпывающее доказательство превосходства подхода: для более строгой оценки требуется сравнение с дополнительными baseline-методами, например с TF-IDF + логистической регрессией, косинусным сходством эмбеддингов или другой моделью ранжирования, обученной на тех же данных.

## 2.3 Бизнес-моделирование AS IS и TO BE рекрутмента с сервисом отбора кандидатов

Бизнес-моделирование процессов рекрутмента позволяет наглядно представить текущее состояние системы (AS IS) и определить, каким образом внедрение автоматизированного сервиса отбора кандидатов на основе обработки естественного языка может изменить процесс (TO BE). Данный подход помогает выявить неэффективности текущего процесса, сформулировать потенциальные улучшения и определить, какие организационные эффекты автоматизации требуют дополнительной проверки на данных.

В модели AS IS традиционного рекрутмента процесс отбора кандидатов обычно включает несколько этапов, которые требуют значительных затрат времени и человеческих ресурсов. На начальных этапах рекрутеры размещают вакансию на различных платформах и получают большой объем откликов. Затем следует процесс ручной фильтрации резюме, при котором специалист по подбору кадров тратит время на прочтение и анализ каждого резюме. Этот этап является наиболее трудоемким, так как он предполагает оценку релевантности кандидатов по нескольким параметрам: опыт работы, образование, навыки и соответствие требованиям вакансии. Ошибки при ручной фильтрации, связанные с человеческим фактором, могут привести к упущению подходящих кандидатов или, наоборот, к приглашению на собеседование кандидатов, которые не соответствуют требованиям [8].

Кроме того, в текущей модели рекрутеры часто сталкиваются с проблемой неконсистентности и неструктурированности информации в резюме. Кандидаты могут по-разному описывать свои навыки и опыт, что затрудняет сравнение их профилей с требованиями вакансии. Все это увеличивает время на принятие решения и может привести к повышению затрат на процесс найма [9].

В модели TO BE, с внедрением автоматизированного сервиса для отбора кандидатов, часть вышеперечисленных проблем может быть снижена за счет применения технологий обработки естественного языка. Система автоматически анализирует большое количество резюме и вакансий, используя такие технологии, как векторизация текста и модели машинного обучения для сопоставления навыков кандидатов с требованиями вакансий. По результатам эксперимента из раздела 2.2 модель BERT показала accuracy 0.75 при контрольном ориентире 0.56 для простого предсказания наиболее частого класса и MCC 0.550 относительно нейтрального значения 0. Это позволяет рассматривать модель как потенциальный инструмент предварительного ранжирования, однако не доказывает полного замещения ручной оценки рекрутера. Время, которое ранее тратили на ручную фильтрацию, может быть сокращено, если автоматическое ранжирование используется как вспомогательный этап, а рекрутеры сосредоточиваются на более высокоуровневых задачах, таких как проведение собеседований и оценка соответствия мягких навыков кандидатов компании [6].

Одним из ключевых преимуществ автоматизированного сервиса является его способность учитывать контекст и семантику текста. Модель, такая как BERT, позволяет не ограничиваться поиском ключевых слов в резюме, а учитывать их смысл в зависимости от контекста. Поэтому такой подход может повысить качество сопоставления кандидатов с вакансиями по сравнению с простыми правилами или ручным просмотром больших массивов откликов, но величина этого эффекта должна подтверждаться отдельными экспериментами с альтернативными baseline-методами. Например, если в вакансии требуется знание определенного инструмента, система может находить кандидатов, которые имеют соответствующий опыт, даже если они описали его различными способами [10].

Кроме того, автоматизация рекрутмента с использованием NLP-сервисов способствует стандартизации процесса и может снизить влияние отдельных ошибок, связанных с человеческим фактором. Сервис может не только фильтровать резюме, но и автоматически ранжировать кандидатов по уровню соответствия вакансии, предоставляя рекрутеру отсортированный список кандидатов. На текущем этапе корректнее рассматривать это как механизм поддержки принятия решений: результат демонстрирует улучшение относительно выбранного контрольного подхода, однако для более строгой оценки требуется сравнение с дополнительными baseline-методами и анализ ошибок модели [5].

Модель TO BE с внедрением автоматизации также повышает прозрачность процесса отбора. Система может хранить все данные по результатам анализа резюме, предоставляя рекрутеру возможность оценить, почему тот или иной кандидат был отобран или отклонен. Это особенно важно в условиях высокого объема откликов, когда ручной анализ может быть недостаточно тщательным.

Таким образом, внедрение автоматизированного сервиса для отбора кандидатов на основе обработки естественного языка может повысить воспроизводимость предварительного отбора и сократить часть ручных операций в процессе рекрутмента. Однако выводы о снижении затрат, ускорении найма и повышении качества подбора должны рассматриваться как гипотезы, требующие проверки на расширенном наборе метрик, включая сравнение с альтернативными алгоритмами и оценку влияния сервиса на реальные решения рекрутеров.

## 2.4 Разработка функциональных требований к целевому сервису

Разработка функциональных требований к целевому сервису является важным этапом процесса создания системы для автоматизации отбора кандидатов на основе обработки естественного языка. Основная задача функциональных требований заключается в описании функциональности, которую должна предоставлять система, чтобы соответствовать поставленным бизнес-целям и удовлетворять запросы пользователей. Для формализации этих требований используются различные подходы, такие как диаграммы UML и BPMN, которые позволяют визуализировать процессы и взаимодействие пользователей с системой.

Основной поток деятельности начинается с этапа публикации вакансии работодателем. Система автоматически получает информацию о вакансии и приступает к сбору резюме кандидатов с различных платформ. Следующим шагом является анализ собранных резюме с использованием методов обработки естественного языка. В этом процессе система извлекает ключевые данные из резюме и сопоставляет их с требованиями вакансии. Результатом этого этапа является формирование списка кандидатов, упорядоченных по уровню соответствия [11].

Для описания требований к системе также важно учитывать нефункциональные аспекты, такие как производительность и масштабируемость. Система должна быть способна обрабатывать большое количество резюме и вакансий за приемлемое время, особенно при большом объеме данных. Кроме того, необходимо предусмотреть интеграцию с внешними системами, такими как платформы поиска работы и базы данных резюме [13].

## 3.2 Выбор и обоснование архитектуры сервиса

При выборе и обосновании архитектуры для сервиса автоматизации отбора кандидатов на основе обработки естественного языка необходимо учитывать требования к масштабируемости, производительности и интеграции с внешними системами. Для данного проекта подходит микросервисная архитектура, которая обеспечивает гибкость и возможность независимого развития отдельных компонентов системы.

На верхнеуровневом уровне система состоит из нескольких ключевых компонентов: сервиса управления резюме, сервиса управления вакансиями, сервиса анализа и сопоставления и сервиса пользователей. Каждый из этих сервисов взаимодействует через REST API, что позволяет обеспечивать гибкость и независимость их разработки и развертывания.

Для более детального описания процессов взаимодействия компонентов системы можно использовать диаграмму последовательностей. Например, процесс сопоставления кандидата с вакансией начинается с вызова сервиса вакансий через REST API. Далее система отправляет запрос к сервису анализа резюме для получения списка подходящих кандидатов. Модуль анализа на основе моделей обработки естественного языка анализирует данные и возвращает список кандидатов, упорядоченных по уровню соответствия, после чего сервис вакансий сохраняет результат и предоставляет его рекрутеру через пользовательский интерфейс [20].

## 4.1 Выбор и обоснование жизненного цикла разработки

При выборе жизненного цикла разработки для сервиса автоматизации отбора кандидатов на основе обработки естественного языка из резюме и вакансий необходимо учитывать сложность проекта, необходимость в гибкости и изменениях на этапе разработки, а также потребность в быстром прототипировании и обратной связи от конечных пользователей.

Для данного проекта обоснован выбор инкрементного подхода к разработке, так как он позволяет разделить процесс на несколько этапов, каждый из которых фокусируется на реализации определенной функциональности. Такой подход предоставляет возможность частичного завершения продукта, что важно для получения своевременной обратной связи и внесения изменений в случае необходимости.

Инкрементная модель позволяет улучшать качество продукта с каждой новой версией, постепенно наращивая функциональность и производительность системы. На первом этапе можно сосредоточиться на базовой функциональности, такой как загрузка и хранение резюме и вакансий, а затем добавлять компоненты семантического анализа и сопоставления, интеграцию с векторными СУБД и другие важные элементы.

Для разработки сервиса автоматизации отбора кандидатов на основе обработки естественного языка из резюме и вакансий необходимо выбрать инструменты, которые обеспечат производительность, масштабируемость и удобство при работе с большими объемами данных.
