\chapter{Реализация и экспериментальная проверка}
\label{chapter:implementation}

\section{Жизненный цикл разработки}

Для разработки выбран инкрементный подход. Он позволяет сначала реализовать минимальный прототип обработки резюме и вакансий, затем последовательно уточнять модель, интерфейсы, хранение данных и экспериментальную проверку. Такой жизненный цикл соответствует исследовательскому характеру работы: требования и качество модели уточняются по мере накопления результатов.

\section{Выбор инструментов}

Основным языком реализации выбран Python, поскольку он имеет развитую экосистему для обработки текстов, машинного обучения, API-сервисов и экспериментальных прототипов. В качестве среды разработки в черновиках выбран PyCharm. Для хранения структурированных данных рассматривается MongoDB, для векторного поиска --- Weaviate. Для API-слоя могут использоваться Django, Flask или FastAPI; окончательный выбор должен быть согласован с фактической реализацией прототипа.

\section{Текущий прототип}

В репозитории присутствуют несколько версий клиент-серверного прототипа. Последняя зафиксированная версия находится в каталоге \texttt{version\_4}. Код сервера и клиента сохранен в приложении в виде листингов, чтобы сборка ПЗ оставалась самодостаточной.

Файл \texttt{weaviate-quickstart/docker-compose.yml} показывает начальную конфигурацию окружения для векторной базы данных. Это подтверждает выбранное направление: хранение эмбеддингов и семантический поиск должны быть вынесены в специализированный компонент.

\section{План экспериментальной проверки}

Экспериментальная проверка должна подтвердить, что модель ранжирования помогает быстрее находить релевантных кандидатов. Минимальный план проверки:
\begin{compactenum}
  \item подготовить обезличенный набор пар <<резюме--вакансия>>;
  \item определить целевой статус, который считается положительным примером;
  \item запустить модель сопоставления для каждой пары;
  \item сравнить ранжирование с фактическими решениями HR-специалистов;
  \item рассчитать метрики качества и описать ошибки модели.
\end{compactenum}

Результаты тестирования должны быть представлены в таблицах и графиках. Для защиты особенно важны не только абсолютные значения метрик, но и анализ случаев, где модель ошибается: нерелевантные ключевые слова, неполные резюме, различия в формулировках должностей и неоднозначные требования вакансий.

\section{Ограничения текущего состояния}

На момент первичного сборочного прохода в репозитории есть рабочие заготовки и черновики, но не хватает финального эксперимента, утвержденных персональных данных, официальных бланков титульной части и полного списка источников. Эти элементы вынесены в чеклист перед сдачей.
