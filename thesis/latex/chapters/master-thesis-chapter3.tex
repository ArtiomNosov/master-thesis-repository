\chapter{Проектирование архитектуры сервиса}
\label{chapter:architecture}

\section{Выбор архитектурного подхода}

Для сервиса автоматизации отбора кандидатов выбран микросервисный подход. Он соответствует структуре задачи: обработка резюме, обработка вакансий, сопоставление кандидатов, хранение данных и пользовательский интерфейс могут развиваться независимо. Такое разделение снижает связность компонентов и позволяет отдельно масштабировать наиболее нагруженные части системы.

На верхнем уровне система включает:
\begin{compactitem}
  \item сервис управления вакансиями;
  \item сервис управления резюме;
  \item сервис NLP-обработки и формирования эмбеддингов;
  \item сервис сопоставления и ранжирования кандидатов;
  \item хранилище структурированных данных;
  \item векторное хранилище для семантического поиска;
  \item клиентское приложение или API для HR-специалиста.
\end{compactitem}

\section{Данные и хранилища}

Структурированные данные о пользователях, вакансиях, резюме и событиях взаимодействия могут храниться в документно-ориентированной базе данных. В проекте рассматривается MongoDB, поскольку она хорошо подходит для хранения JSON-подобных документов и гибких схем \cite{MongoDBDocs}.

Векторные представления вакансий и резюме целесообразно хранить отдельно, в базе данных, оптимизированной для поиска по близости. В текущем репозитории уже есть quickstart для Weaviate, что делает эту систему естественным кандидатом для прототипа.

\section{Интерфейсы компонентов}

Компоненты сервиса должны обмениваться данными через явные API-контракты. Для первой версии достаточно REST-интерфейса:
\begin{compactitem}
  \item создание и обновление вакансии;
  \item загрузка резюме;
  \item запуск пересчета эмбеддингов;
  \item получение ранжированного списка кандидатов;
  \item получение подробностей по конкретной паре <<резюме--вакансия>>.
\end{compactitem}

В дальнейшем API может быть расширен событиями для асинхронной обработки: например, новая вакансия или новое резюме могут порождать задачу на пересчет эмбеддингов.

\section{Ограничения проектирования}

Основные риски архитектуры связаны с обработкой персональных данных, качеством исходных текстов и объяснимостью результата. Система не должна полностью заменять решение специалиста: на первом этапе она является инструментом приоритизации и подсказки. Итоговое решение о кандидате остается за пользователем.
