## Слайд 2: Введение и проблематика

### [ НА СЛАЙДЕ ]

**Проблема:** первичный скрининг кандидатов — самый трудоёмкий этап рекрутмента

| Подход | Ограничение |
|---|---|
| Ручной отбор | 45 мин на цикл скрининга из 20 заявок |
| TF-IDF / BM25 | Нет семантики: «Java-разработчик» ≠ «backend engineer» |
| LLM-скоринг для всего пула | Высокая стоимость при большом числе кандидатов |

**Position Bias** — рекрутер предпочитает кандидатов из начала списка, а не лучших по качеству

🖼 **ИЗОБРАЖЕНИЕ:** воронка подбора с выделенным узким местом (скрининг)
*→ нарисовать схему: Отклики → [⚠ Скрининг — узкое место] → Интервью → Оффер*

---

### [ ГОВОРЮ ]

> Проблема, которую я решаю, хорошо известна HR-специалистам: первичный скрининг — это бутылочное горлышко всей воронки подбора. Для расчета принято 45 минут ручного просмотра на цикл из 20 заявок, то есть 2,25 минуты на одну заявку. Классические текстовые методы — TF-IDF и BM25 — не улавливают смысловых связей: они не понимают, что «Java-разработчик» и «backend engineer» — это один и тот же профиль. Дорогой LLM-скоринг решает эту проблему, но применять его к каждому из сотни кандидатов экономически нецелесообразно. Дополнительно исследования показывают эффект position bias: без объективного ранжирования рекрутер систематически отдаёт предпочтение тем, кто оказался первым в списке.

Ключевая проблема — это шаг ручного просмотра: он не масштабируется, субъективен и зависит от порядка, в котором резюме лежат в базе. Вот с этим и работает моя диссертация.

Рекрутер больше не видит резюме в произвольном порядке — он видит топ-N наиболее релевантных. При необходимости для этого топа можно дополнительно вызвать дорогой LLM-скоринг — но уже не для ста кандидатов, а для пяти-десяти.

## Слайд 5: Обзор методов и обоснование выбора

### [ НА СЛАЙДЕ ]

**Эволюция методов семантического сопоставления текстов:**

| Метод | Семантика | Скорость инференса | Итог |
|---|---|---|---|
| TF-IDF / BM25 | ✗ | O(1) | Не подходит |
| Word2Vec / FastText | Частичная | Быстро | Слабое качество |
| BERT Cross-Encoder | ✓ | O(N) — медленно на больших базах | Не масштабируется |
| **Bi-Encoder (Sentence-BERT)** | **✓** | **O(1) после индексации** | **✓ Выбрано** |

**Базовая модель:** `cointegrated/rubert-tiny2`
- 29 млн параметров · русскоязычная · работает без GPU

🖼 **ИЗОБРАЖЕНИЕ:** схема bi-encoder vs cross-encoder
*→ нарисовать: два отдельных энкодера + cosine similarity слева; один совместный энкодер справа*

---

### [ ГОВОРЮ ]

> Я рассматривал четыре класса методов. TF-IDF и BM25 быстрые, но не понимают смысла. Word2Vec даёт частичную семантику, но качество недостаточное для HR-задачи. Cross-encoder — полноценный BERT, который обрабатывает пару «вакансия + резюме» совместно — даёт высокое качество, но при базе в тысячу резюме требует тысячи обращений к модели на каждый запрос, что неприемлемо по скорости. Bi-encoder решает эту проблему: вакансия и резюме кодируются независимо, резюме индексируются один раз, а ранжирование — это просто косинусное сходство между векторами, что занимает миллисекунды. В качестве базовой модели я выбрал `cointegrated/rubert-tiny2` — это лёгкая русскоязычная модель на 29 миллионов параметров, которая разворачивается на обычном CPU-сервере без GPU.

## Слайд 7: Архитектура интеграции в ATS Reqcore

### [ НА СЛАЙДЕ ]

🖼 **ИЗОБРАЖЕНИЕ:** скриншот настроек AI в Reqcore — режим biencoder
*→ файл:* `experiments/results/ai_settings_biencoder.png`

🖼 **ИЗОБРАЖЕНИЕ:** страница вакансии с ранжированными кандидатами
*→ файл:* `experiments/results/job_page_priya_biencoder.png`

**Два режима скоринга в Reqcore:**

```
Reqcore backend (TypeScript)
  ├── provider = 'biencoder' → FastAPI ML-сервис (локально)
  └── provider = 'llm'       → OpenAI API (платно)
```

**FastAPI эндпоинты:**
- `POST /index/candidate` — индексировать резюме
- `POST /search/rank` — ранжировать по тексту вакансии

---

### [ ГОВОРЮ ]

> Микросервис написан на Python с FastAPI и разворачивается рядом с основным бекендом Reqcore. В системе появился роутер скоринга: если администратор выбирает режим `biencoder`, запросы ранжирования идут в локальный ML-сервис. Если выбран режим `llm` — вызывается OpenAI. На скриншотах видно реальный интерфейс: в настройках выбран biencoder, а на странице вакансии кандидаты отсортированы по cosine similarity. Важно: переключение между режимами не требует изменения кода — только смена конфигурации в административном интерфейсе.

## Слайд 11: Планы развития (Future Work)

### [ НА СЛАЙДЕ ]

**Масштабирование:**
- Vector DB (Qdrant / Milvus) — индексация миллионов профилей на диске
- ANN (HNSW / IVF) — субмиллисекундный поиск на больших базах

**Качество:**
- Learning-to-Rank (уровень 2) — RankNet / LambdaMART
- Hard Negative Mining при обучении

**Справедливость и объяснимость:**
- Fair Ranking — аудит на предубеждения
- Explainability — какие навыки влияют на скор

**Данные:**
- Мультиязычность (multilingual-e5)
- Дообучение на hh.ru / Работа.ру

---

### [ ГОВОРЮ ]

> Текущая реализация — это in-memory индекс, который хорошо работает при базе до нескольких тысяч резюме. Для промышленного масштаба нужна vector database — например, Qdrant или Milvus — с алгоритмами приближённого поиска HNSW. Качество ранжирования можно улучшить добавлением второго уровня — модели Learning-to-Rank, которая учтёт структурированные атрибуты вроде опыта и образования. Важный вопрос, который я планирую исследовать — справедливость ранжирования: нет ли у модели демографических предубеждений, унаследованных из обучающих данных.
